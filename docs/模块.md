# 模块说明

> 本文描述当前主线模块，而不是历史上出现过的全部能力。

## 1. 总体架构

当前系统按四层组织：

1. 交互层：CLI（`webnovel.py`）+ Dashboard（FastAPI + React）
2. 运行层：会话、索引、RAG 验证、运行时迁移
3. 数据层：`state.json / index.db / vectors.db` + `.webnovel/codex/*`
4. 质量层：一致性检查、RAG benchmark、CI 发布门禁

## 2. 主线模块清单

| 模块 | 代码位置 | 主要职责 | 关键输出 |
|---|---|---|---|
| CLI Gateway | `webnovel-writer/scripts/data_modules/webnovel.py` | 统一命令入口、参数解析、转发子模块 | 标准化命令调用链 |
| Codex CLI | `webnovel-writer/scripts/data_modules/codex_cli.py` | 会话管理、索引状态、RAG 验证 | 统一 `codex` 子命令组 |
| Session Manager | `webnovel-writer/scripts/data_modules/session_manager.py` | 创建 / 清理会话目录与 profile | `.webnovel/codex/sessions/*` |
| Incremental Indexer | `webnovel-writer/scripts/data_modules/incremental_indexer.py` | 生成索引产物与索引状态 | `fast-index.json`、`index-state.json` |
| RAG Verifier | `webnovel-writer/scripts/data_modules/rag_verifier.py` | 读取 benchmark 产物并执行 RAG 门禁 | `rag-benchmark.json` 验证报告 |
| Runtime | `webnovel-writer/dashboard/routers/runtime.py` + `dashboard/services/runtime/` | 运行时画像、迁移预览与迁移执行 | `.webnovel/migrations/*` |
| Read Model API | `webnovel-writer/dashboard/app.py` | 实体、章节、图谱、统计、文件读取、SSE | Dashboard 只读视图数据 |
| Frontend Pages | `webnovel-writer/dashboard/frontend/src/pages/*` | 只读页面编排与状态展示 | 用户观察入口 |
| Release Gate | `.github/workflows/app-release-gate.yml` | 在 CI 中执行 `rag verify` 门禁 | 发布阻断结果 |

## 3. 当前主线流程

### 3.1 CLI / Codex 写作流程

1. `webnovel.py codex session start --profile ...`
2. Codex 直接编辑小说项目文件
3. `webnovel.py codex index status`
4. `webnovel.py codex rag verify`
5. `webnovel.py codex session stop --session-id ...`

### 3.2 Dashboard 观察流程

1. 启动 `dashboard.server`
2. 可选启用 Basic Auth
3. 前端读取只读 API、文件树、统计与事件流
4. 作者通过 CLI / Codex 完成写操作，而不是通过 Dashboard HTTP 写接口

### 3.3 发布门禁流程

1. CI 生成最小 benchmark fixture 项目
2. 执行 `webnovel.py codex rag verify --report json`
3. 验证失败则 workflow 失败

## 4. 历史能力说明

仓库中仍可能保留部分历史服务实现、旧测试或旧前端组件，例如：

- Skills CRUD
- Settings Dictionary 写流程
- Outlines split / resplit
- Edit Assist
- Codex Bridge 写作辅助链路

这些内容不应被理解为当前主线产品的现行承诺；只有当路由挂载、文档和测试同时恢复一致时，才应重新视为支持能力。

## 5. 已知限制

1. `rag verify` 已 fail-closed，并接入 CI，但正确性 / 性能值仍依赖真实 benchmark 产物质量。
2. 当前内置认证是最小 Basic Auth 闸门，适合单租户 / 受信任环境，不等于完整账户系统。
3. Dashboard 仍然是只读观察面，不是完整写作工作台。
